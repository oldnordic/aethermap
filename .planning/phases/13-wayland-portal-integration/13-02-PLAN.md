  <name>Create focus_tracker.rs module with FocusTracker and FocusEvent</name>

  <action>
Create new file razermapper/razermapper-gui/src/focus_tracker.rs with:

1. FocusEvent struct:
   ```rust
   #[derive(Debug, Clone, Serialize, Deserialize)]
   pub struct FocusEvent {
       pub app_id: String,        // e.g., "org.alacritty", "firefox"
       pub window_title: Option<String>,  // May be empty on some compositors
   }
   ```

2. FocusTracker struct:
   ```rust
   pub struct FocusTracker {
       portal: Option<ashpd::desktop::window::WindowProxy>,
       running: Arc<AtomicBool>,
   }
   ```

3. FocusTracker implementation:
   - new() - Creates tracker, attempts to initialize portal
   - start() - Spawns async task that listens for focus changes
   - stop() - Sets running flag to false
   - handle_portal_error() - Logs warnings when portal unavailable

4. Portal integration using ashpd 0.8:
   - Use ashpd::desktop::window::WindowProxy for focus tracking
   - Request window focus monitoring via portal API
   - Stream focus events via receive_focus_change() or similar
   - Extract app_id from focus change events

5. Graceful fallback:
   - If portal initialization fails, log warning and continue
   - Return None from new() if portal unavailable
   - FocusTracker methods should be no-ops when portal is None

Key design decisions:
- Portal permission request happens on first focus event (seamless UX)
- app_id is the primary identifier (window_title may be empty/missing)
- Running on Wayland without portal support results in graceful degradation
- Async task runs until stop() called or GUI exits

Note: ashpd 0.8 API specifics - use WindowProxy::receive_focus_change() stream
for monitoring focus changes. The stream emits (app_id, window_title) tuples.

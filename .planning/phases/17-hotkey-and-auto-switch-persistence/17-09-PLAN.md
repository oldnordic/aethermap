# Phase 17 Plan 09: GlobalHotkeyManager Startup Initialization

## Objective

Initialize GlobalHotkeyManager during daemon startup and load hotkey bindings from device_profiles.yaml so that hotkeys are active immediately after restart, without requiring user GUI interaction.

**Purpose:** Gap closure - GlobalHotkeyManager::load_bindings() exists and is called by IPC handlers after config changes, but there's no explicit startup initialization call in main.rs. This means hotkeys don't work until the first time a user interacts via the GUI.

**Output:** Hotkey bindings loaded and active at daemon startup.

## Background from VERIFICATION.md

**Gap 1: GlobalHotkeyManager Startup Initialization (Medium Priority)**

- Issue: `load_bindings()` exists but no startup call in main.rs or ipc.rs::start
- Impact: Hotkeys work after first GUI interaction, not immediately after restart
- Fix required: Initialize GlobalHotkeyManager and call load_bindings() during daemon startup
- Artifacts: razermapper/razermapperd/src/global_hotkey_manager.rs:128, razermapper/razermapperd/src/main.rs

The GlobalHotkeyManager::load_bindings() method at line 128 has a substantive implementation:
- Reads bindings from ConfigManager::get_all_hotkey_bindings()
- Normalizes modifier names for case-insensitive matching
- Falls back to default_hotkey_bindings() if none configured
- Logs the number of bindings loaded

However, this is only called from IPC handlers (RegisterHotkey/RemoveHotkey in ipc.rs), not during daemon startup.

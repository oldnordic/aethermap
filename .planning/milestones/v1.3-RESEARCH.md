## Summary

This milestone adds two major feature areas: Wayland compatibility improvements and advanced analog stick processing. For Wayland, we need to integrate with xdg-desktop-portal for window focus detection (enabling auto-profile switching) and ensure global hotkeys work on Wayland compositors. For analog sticks, we need multiple processing modes (gamepad emulation, WASD output, mouse control, camera/scroll) and a full calibration GUI.

**Primary recommendation:** Use ashpd (Rust xdg-desktop-portal bindings) for Wayland integration, extend existing AnalogProcessor from Phase 11 with mode-based processing, and build calibration UI using existing Iced patterns.

## Architecture Patterns

### Wayland Focus Detection

**Pattern: Portal Proxy with Event Stream**

```rust
// Portal client creates a watch on window focus changes
pub struct WaylandFocusClient {
    portal: ashpd::desktop::window_identifier::WindowIdentifier,
    focus_tx: mpsc::Sender<FocusEvent>,
}

// Daemon subscribes to focus events
pub struct AutoProfileSwitcher {
    focus_rx: mpsc::Receiver<FocusEvent>,
    device_profiles: HashMap<DeviceId, Vec<ProfileConfig>>,
}
```

**Key considerations:**
- Portal requires user approval (first-use permission prompt)
- Focus events only for windows that registered interest
- Fallback to manual switching if portal unavailable
- Daemon must run as root, but GUI runs as user → IPC bridge needed

**Auto-profile switching workflow:**
1. GUI (running as user) requests window focus via portal
2. GUI sends focus updates to daemon via IPC
3. Daemon matches window/app to configured profile
4. Daemon switches active profile for configured devices

### Analog Processing Modes

**Pattern: Strategy Enum for Output Modes**

```rust
pub enum AnalogMode {
    Gamepad,    // Xbox 360 controller emulation
    WASD,       // Eight-way key output (WASD + diagonals)
    Mouse,      // Relative cursor movement
    Camera,     // Scroll or camera rotation events
}

pub struct AnalogProcessor {
    mode: AnalogMode,
    calibration: AnalogCalibration,
    // ... per-layer state
}

impl AnalogProcessor {
    pub fn process(&mut self, x: i32, y: i32) -> Vec<InputEvent> {
        match self.mode {
            AnalogMode::Gamepad => self.to_gamepad_events(x, y),
            AnalogMode::WASD => self.to_wasd_events(x, y),
            AnalogMode::Mouse => self.to_mouse_events(x, y),
            AnalogMode::Camera => self.to_scroll_events(x, y),
        }
    }
}
```

// uinput device setup for gamepad
fn create_xbox360_virtual_device() -> Result<VirtualDevice, Error> {
    VirtualDevice::builder(&name)?
        .enable_event_type(EventType::ABSOLUTE)

        .enable_abs_axis(AbsoluteAxisType::ABS_Z, AbsInfo::default())  // Right stick X
        .enable_abs_axis(AbsoluteAxisType::ABS_RZ, AbsInfo::default()) // Right stick Y

        .enable_key(Key::BTN_SOUTH)   // A button
        .enable_key(Key::BTN_EAST)    // B button
        .enable_key(Key::BTN_NORTH)   // X button
        .enable_key(Key::BTN_WEST)    // Y button

        .enable_key(Key::BTN_TR)      // Right bumper

## Implementation Details

### Wayland Focus Detection

**Phase 13: Wayland Portal Integration**
1. Add ashpd dependency to GUI (not daemon - daemon is root, portal is user)
2. Create FocusTracker in GUI using ashpd's WindowIdentifier
3. Add IPC message: `FocusChanged { app_id: String, window_title: Option<String> }`
4. Create AutoProfileSwitcher in daemon matching focus → profile rules
5. YAML config extension: `auto_switch_rules: [{ app_id: "firefox", profile: "browser" }]`

**Challenges:**
- Portal requires D-Bus session bus (may not be available in all Wayland sessions)
- Some compositors implement portal partially (graceful degradation needed)
- GUI needs to send focus updates while daemon runs as root → existing IPC bridge

### Global Hotkeys on Wayland

**Challenge:** Wayland doesn't allow global key grabs (security feature)

**Solution approaches:**
1. **Wayland protocol shortcuts API:** Implement org.wayland.shortcuts (limited compositor support)
2. **Portal-based global shortcuts:** xdg-desktop-portal-global-shortcuts (newer, limited adoption)
3. **libinput hook at daemon level:** Since daemon runs as root with /dev/input access, can read all input

**Recommended:** libinput hook (daemon already grabs devices) + IPC broadcast for hotkey-triggered profile switch

### Analog Processing Modes

**WASD Mode (8-way digital output):**
- Divide analog space into 8 sectors (N, NE, E, SE, S, SW, W, NW)
- Deadzone: circular zone at center = no output
- Beyond deadzone: determine sector, send corresponding key(s)
- Diagonals: press two keys (e.g., NE = W + D)
- Release: when re-entering deadzone or changing sectors

**Mouse Mode:**
- Analog distance from center → mouse velocity
- Sensitivity: pixels per tick
- Acceleration: optional curve for faster movement at full deflection
- uinput EV_REL events for REL_X, REL_Y

**Camera/Scroll Mode:**
- Similar to mouse mode but different target axis
- Option 1: REL_WHEEL events for scrolling
- Option 2: Key repetition for camera keys (PageUp/Down, arrow keys)
- Configurable axis mapping

**Gamepad Mode:**
- Direct pass-through: analog stick position → gamepad ABS_X/ABS_Y
- Calibration applies before output (deadzone, sensitivity, range)
- Creates virtual Xbox 360 device via uinput
- Games see standard gamepad (works with any gamepad-enabled software)

## Implications for Roadmap

**Phase dependencies:**
- Phase 13 (Wayland) is independent - can proceed in parallel with v1.2 completion
- Phases 14-16 (Analog) build on Phase 11's foundation but extend significantly
- Phase 14 should precede 15-16 (gamepad mode provides testing foundation)

**Parallel execution opportunities:**
- Phase 13 (Wayland GUI) and Phase 14 (Gamepad mode) have no dependencies
- Phase 15 and 16 can have parallel waves within each phase

**Risks:**
- Wayland portal support varies by compositor - need graceful degradation
- Gamepad emulation may require specific uinput device configuration - testing on real hardware needed
- Global hotkeys on Wayland are still evolving - may need compositor-specific workarounds

## Open Questions for User

1. **Portal permission UX:** How should GUI handle first-use portal permission prompt? (Show instructions, auto-trigger on startup, config option?)

2. **Profile priority:** When both auto-switch (focus) and manual switch (hotkey) occur, which wins? (Recommend: manual overrides auto until focus changes)

3. **WASD diagonal behavior:** Should diagonals be modifiable? (e.g., some users want diagonal to be different key combination)

4. **Gamepad mapping extent:** Should we support remapping gamepad buttons too, or just pass through analog? (Recommend: Phase 14 = pass-through only, future = full gamepad remapping)

5. **Camera mode default:** Should default be scroll or key-based camera control? (Recommend: scroll, simpler for most users)
